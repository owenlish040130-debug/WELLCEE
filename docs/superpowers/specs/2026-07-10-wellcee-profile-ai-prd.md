## 一、产品背景与目标

### 1.1 产品定位

Wellcee（唯心所寓）是无中介 C2C 个人直租 + 国际化租房社交平台。用户在这里找房、找室友、出租转租、参与社区生活。

### 1.2 当前问题

现有个人资料完善流程是老式表单填写，用户填写意愿低、资料完整度参差不齐。室友匹配、找房推荐、信任建设等下游功能缺乏高质量的结构化用户数据。

### 1.3 项目目标

结合 AI 大模型，重新设计「登录后 → 个人主页 → 编辑资料」流程：

| 目标 | 衡量指标 |
|------|---------|
| **降低填写摩擦** | 资料编辑完成率提升、平均填写时长缩短 |
| **提升资料质量** | 标签数量增加、标签准确率 > 90% |
| **增强个性化表达** | 用户留存简介原创度 > 80% |
| **支撑下游匹配** | 有效室友匹配标签覆盖率 > 85% |

### 1.4 AI 核心价值定位

- **效率**：减少用户打字和决策成本（规则预填 + AI 追问 + 一键生成简介）
- **深度**：帮用户表达出「自己说不出来但确实存在」的偏好和画像

**AI 的定位是「思考搭档」，不是「代写」。**

## 二、用户流程

### 2.1 完整流程（7 步）

```
Step 1: 固定表单
    ├─ 头像（图片上传）
    ├─ 昵称（文本输入）
    ├─ 年龄（出生年份选择器）
    ├─ 职业（文本输入）
    ├─ 地区（选择/输入）
    ├─ 语言（多选添加）
    └─ 当前需求（多选卡片）
        ├─ 🏠 我要出租（整租/单间）
        ├─ 🔄 我要转租（整租/单间）
        ├─ 🤝 找室友-已有房
        ├─ 🤝 找室友-还没房
        └─ 👀 暂无需求，逛逛社区
            ↓
Step 2: 自由文本输入
    ├─ 引导文案根据需求栏动态切换（见 3.1）
    └─ 支持文字输入 / 语音输入（可选）
            ↓
Step 3: AI 追问（用户主动唤起）
    ├─ 用户点击「✨ AI 帮我看看」
    ├─ AI 展示 1-3 条追问建议（每条带 2-4 个选项 + 自己写）
    ├─ 用户操作：采纳 / 单条跳过 / 全部跳过
    └─ 采纳的回答自动插入正文
            ↓
Step 4: AI 提取标签
    ├─ 通用标签（性格 + 爱好）—— 所有人
    ├─ 需求标签（室友期望 / 求租偏好 / 房源特征）—— 按需
    └─ 每个标签标注来源（free_text / question_answer）和置信度
            ↓
Step 5: 用户确认 / 调整标签
    ├─ 标签按维度分类展示（性格 / 爱好 / 室友期望 / 求租偏好 / 房源特征）
    ├─ 可删除 / 手动添加 / 修改标签
    └─ 支持回退到 Step 3 补充追问
            ↓
Step 6: AI 生成个人简介
    ├─ 基于原文 + 确认标签，不编造事实
    └─ 根据需求栏自动选择简介视角（室友版 / 房东版 / 通用版）
            ↓
Step 7: 完成 ✨
    └─ 保存到个人资料，实时生效
```

### 2.2 用户路径自由度

- 固定表单 → 自由文本：线性的，必须先填表单
- 自由文本 → AI 追问：**完全由用户控制**，不点 ✨ 按钮 AI 绝不主动出现
- 追问环节：可循环多次（每次最多 3 条追问，总轮数前端限制 5 轮后轻提示）
- 标签 → 简介：标签确认后才能生成简介
- 全程可随时保存退出，已确认的字段不丢失

## 三、功能需求

### 3.1 差异化输入引导

自由文本区的 placeholder 根据需求栏动态切换：

| 用户需求 | 输入框引导文案 |
|---------|--------------|
| 仅「暂无需求」 | *介绍你自己——你的性格、爱好、生活方式…* |
| 「找室友」 | *介绍你自己，以及你理想的室友是什么样的——作息、生活习惯、你们会怎么相处…* |
| 「求租」 | *介绍你自己，以及你想找什么样的房子——区域、预算、你对居住环境最在意的是什么…* |
| 「出租」 | *介绍你自己，以及你出租的房子是什么样的——位置、户型、周边配套、有什么特别值得说的…* |
| 「转租」 | *介绍你自己、为什么转租、房子怎么样、你希望找什么样的接手人…* |
| 多选（如 找室友+求租） | *介绍你自己、你想找什么样的室友、以及你对房子的期望…* |

### 3.2 AI 追问机制

#### 3.2.1 触发规则

- **唯一触发方式**：用户点击「✨ AI 帮我看看」
- AI 不自动弹出、不在输入时插话
- 停止输入超过 15 秒 + 已输入 > 100 字 → 出现轻声提示「💡 有追问待查看」（不展开内容）

#### 3.2.2 追问策略

| 策略 | 触发条件 | 示例 |
|------|---------|------|
| 具象化 | 检测到模糊标签（如「运动」「音乐」无具体类型） | 「💡『运动』具体是什么类型？」 |
| 场景化 | 检测到可展开的生活习惯（如「晚睡」「安静」） | 「💡 说到晚睡——你对夜间噪音的态度是？」 |
| 画像补充 | 需求类型的关键匹配维度尚未覆盖 | 「💡 你对合租清洁的标准大概是？」 |

#### 3.2.3 退出机制（三级）

| 层级 | 操作 | 后果 |
|------|------|------|
| 单条跳过 | 点某条建议的「跳过」 | 该条消失，其他建议保留 |
| 本轮全跳 | 点「全部跳过」 | 本次所有建议消失，不影响后续使用 |
| 隐式静默 | 连续 3 次点「全部跳过」 | 后续自动转为「仅显示轻提示、不自动展开」，用户仍可手动展开 |

#### 3.2.4 设计约束

- 每轮最多 3 条追问
- 每条追问必须附带 2-4 个可点击选项 + 1 个「自己写」
- 已回答过的维度不再追问
- 追问不脱离租房/合租场景

### 3.3 标签体系

#### 3.3.1 三层架构

```
标签体系
├── 第一层：通用标签（所有用户提取）
│   ├── 性格标签：MBTI、安静/外向、计划型/随性型、独立/合群…
│   └── 爱好标签：运动类、宅家类、社交类、文艺类…
│
├── 第二层：需求标签（根据需求栏触发）
│   ├── 找室友 → 室友期望：作息兼容度、清洁标准、社交频率、宠物态度、访客态度、厨房共享态度、噪音容忍度…
│   ├── 求租 → 求租偏好：目标区域、预算区间、房型、交通要求、宠物友好、入住时间…
│   ├── 出租 → 房源特征：位置、户型、装修水平、楼层、周边配套、租金…
│   └── 转租 → 转租原因 + 房源特征（同出租）
│
└── 第三层：文化特色标签（二期）
    └── 五行属性（金/木/水/火/土）：从出生年份自动计算，标注为文化特色
```

#### 3.3.2 标签属性

每个标签携带：

```json
{
  "label": "安静型",
  "dimension": "personality",
  "source": "free_text | question_answer",
  "confidence": 0.85,
  "roommate_relevance": "high | medium | low"
}
```

- `source`：标签来源，前端展示差异化 UI（追问得来标注 💬）
- `confidence`：为下游匹配算法提供权重参考
- `roommate_relevance`：标记该标签对室友匹配的影响程度

### 3.4 简介生成规则

- **铁律**：只基于用户提供的原文和确认标签进行组织润色，不添加用户没有说过的事实
- 语气自然口语化，保持用户原文的语言风格
- 根据需求类型自动调整收尾语气（找室友/求租/出租/转租/通用）

### 3.5 MBTI 处理

- 不放在固定表单中
- AI 不从文本推断 MBTI
- 用户在自由文本中提到了 → 直接提取为标签
- 没提到 → 追问环节询问（附带简短选项帮助不知道 MBTI 的用户）

### 3.6 需求栏设计（不做硬性身份区分）

不做「房东/租客」二选一，因为 Wellcee 用户的真实行为是复合的——正在转租的人同时可能在找房。需求栏是一个**可变状态标识**，不是永久身份。

## 四、交互设计要点

### 4.1 追问面板

- 追问以气泡形式出现在输入框上方
- 轻微淡入动效，不打断输入流
- 用户选择了选项 → 文本直接插入到光标位置
- 用户 10 秒不操作 → 气泡收起为一行小字

### 4.2 标签确认面板

- 按维度分区域展示（性格 / 爱好 / 室友期望 / 求租偏好 / 房源特征）
- 每个标签可 × 删除、可点击编辑、可拖动调整
- 底部提供「+ 手动添加标签」和「🔄 AI 追问补充」按钮
- 顶部显示标签总数和覆盖维度数

### 4.3 简介预览

- 简介生成后以卡片形式预览
- 提供「复制」「修改」「重新生成」操作
- 「重新生成」允许用户调整语气偏好（更活泼 / 更正式 / 更简洁）

## 五、信任建设（二期）

### 5.1 匹配透明度

浏览其他用户资料时，展示：
- 共同点高亮：「你们有 4 个共同点」
- 差异点提示：「有 1 个不同点：他偏好安静，你偏好社交」

把信任从「这个人可不可信」的模糊判断，转化为「我们在这些具体维度上合不合适」的客观对比。

## 六、非功能需求

| 维度 | 要求 |
|------|------|
| AI 追问响应延迟 | < 3 秒（P95） |
| 标签提取延迟 | < 2 秒（P95） |
| 简介生成延迟 | < 5 秒（P95） |
| 单次流程 Token 成本 | < ¥0.01 |
| 标签提取准确率 | > 90%（人工抽检） |
| 简介生成可用率 | > 95%（用户不手动大幅修改） |
| 支持语言 | 中文 + 英文（二期） |

## 八、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| AI 追问让用户觉得烦 | 放弃填写 | 完全用户主动唤起 + 三级退出 + 隐式静默 |
| 标签提取不准 | 匹配失效 | 标签确认环节（用户兜底）+ source/confidence 标记 |
| LLM 服务不稳定 | 功能不可用 | 三级降级链（主线→备线→兜底） |
| 多语言用户输入混用 | 提取错误 | 一期仅中文，二期加英文 |
| Token 成本超预算 | 运营压力 | Prompt Cache + 错峰优惠 + 降级备选模型 |
